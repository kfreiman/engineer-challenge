# ADR-005: Структура Go-приложения на основе DDD-lite и Clean Architecture

**Статус:** Accepted  
**Дата:** 2026-03-09  
**Автор:** kfreiman

## 1. Контекст

Проекту требуется заложить масштабируемую, удобную в использовании и поддержке структуру кодовой базы. Приложение предполагается развивать длительное время, поэтому важно обеспечить независимость бизнес-логики от деталей реализации (баз данных, транспортных протоколов).

**Ограничения:**

- Необходимость баланса между сложностью архитектуры и скоростью разработки.
- Структура должна быть понятна Go-разработчикам и не превращаться в "Java-style" с избыточными абстракциями.
- Использовать минимальные примеры.

## 2. Принятое решение

Использовать подход **DDD Lite, CQRS и Clean Architecture**.
Бизнес-логика будет инкапсулирована в доменном слое с использованием богатой (behavior-oriented) модели.

## 3. Технические детали и механизмы

Проект будет разделен по ограниченным контекстам (Bounded Contexts). Внутри каждого контекста соблюдается следующая структура:

```text
engineer-challenge/
├── cmd/                          # Entry points 
├── internal/
│   ├── [context]/                # Ограниченный контекст (например, billing)
│   │   ├── domain/              # Domain Layer — бизнес-логика
│   │   │   ├── entity/         # Сущности
│   │   │   ├── valueobject/    # Value Objects
│   │   │   ├── service/        # Domain Services
│   │   │   └── repository.go   # Интерфейсы репозиториев
│   │   │
│   │   ├── app/                 # Application Layer
│   │   │   ├── command/        # Сценарии изменения состояния (Commands)
│   │   │   └── query/          # Сценарии чтения (Queries)
│   │   │
│   │   ├── adapters/            # Infrastructure Layer (реализации)
│   │   │   ├── *_repository.go # Реализации репозиториев
│   │   │   └── *_client.go     # Внешние клиенты
│   │   │
│   │   ├── ports/               # Входные точки приложения
│   │   │   ├── http.go         # HTTP handlers
│   │   │   └── grpc.go         # gRPC handlers
│   │   │
│   │   └── service/             # Сборка Application
│   │       └── application.go  # Wiring всех компонентов
│   │
│   └── [...другие контексты...]
│
├── api/                          # API спецификации (OpenAPI, protobuf)
└── main.go                       # Entry point
```

**Ключевые принципы:**

1. **Отсутствие тегов в Domain:** Сущности в `domain` не содержат тегов `json:`, `db:` или `gorm:`.
2. **Инкапсуляция и Always-Valid State:** Поля доменных моделей приватные, состояние меняется только через методы, отражающие бизнес-процессы (например, `Reschedule()`, а не `SetStatus()`).
3. **Паттерн Репозиторий:** Интерфейсы определяются в слое `app` (или `domain`), а реализуются в `adapters`.

## 4. Рассмотренные альтернативы

### 4.1. Flat Layout (пакеты по типам: models, services, controllers)

```text
internal/
├── models/       # Все структуры
├── services/     # Вся бизнес-логика
├── handlers/     # Все обработчики
└── repositories/ # Все репозитории
```

**Плюсы:**

- Простота для очень маленьких проектов
- Меньше файлов для начала

**Минусы:**

- Быстро превращается в "спагетти"
- Цикличные зависимости между пакетами
- Сложность навигации по коду
- Нет изоляции между доменами

**Решение:** Отклонено — не подходит для масштабируемой системы.

### 4.2. Полный DDD с событиями домена

```text
internal/
└── domain/
    ├── events/       # Domain Events
    ├── aggregates/   # Агрегаты
    ├── valueobjects/ # Value Objects
    └── services/     # Domain Services
```

**Плюсы:**

- Максимальная строгость и выразительность
- Идеально для сложных бизнес-правил

**Минусы:**

- Высокий порог входа для команды
- Большое количество boilerplate
- Избыточность для простых CRUD операций

**Решение:** Выбран DDD-lite (упрощённая версия) как баланс строгости и скорости.

### 4.3. Чистый Go с пакетами по слоям (pkg/)

```text
pkg/
├── domain/
├── application/
├── infrastructure/
└── interfaces/
```

**Плюсы:**

- Классическая Clean Architecture

**Минусы:**

- Не выделяет Bounded Contexts
- Сложнее поддерживать при росте
- Смешивание контекстов в одном слое

**Решение:** Отклонено в пользу структуры с Bounded Contexts.

## 5. Последствия и риски

### Положительные (Pros)

- Строгая изоляция бизнес-логики позволяет писать быстрые юнит-тесты без мокирования базы данных.
- Код лучше отражает язык предметной области (Ubiquitous Language).
- Легко заменять инфраструктуру. Например, замена PostgreSQL на Firestore затронет только слой `adapters`.

### Отрицательные / Риски (Cons/Risks)

- Увеличенный порог входа для новых разработчиков, не знакомых с CQRS и DDD.
- Бойлерплейт-код для простых CRUD-операций.
- **Митигация:** Использовать генерацию кода; в простых Read-сценариях слоя `app/query` разрешается обращаться напрямую в базу, минуя `domain`.
