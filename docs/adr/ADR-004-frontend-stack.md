# ADR-004: Выбор технологического стека фронтенда для личного кабинета необанка

**Статус:** Accepted  
**Дата:** 2026-03-09  
**Автор:** kfreiman

## 1. Контекст

Мы приступаем к разработке интерфейса личного кабинета необанка. Основной функционал на данном этапе включает предоставление курсов валют для обмена и отображение других финансовых продуктов. Требуется выбрать современный, надежный и поддерживаемый технологический стек, который позволит быстро запускать фичи, опираясь на текущую экспертизу команды, и при этом не будет перегружен излишними архитектурными решениями на старте.

**Ограничения и условия:**

- Команда обладает экспертизой в React SPA (Vite, Tanstack Router, Tanstack Query).
- API бэкенда использует GraphQL.
- Необходим инструмент для генерации типов и хуков для GraphQL.
- Авторизация и аутентификация интегрируются с компонентами `@ory/elements` для снижения бойлерплейта.
- Ограничение на использование внешнего глобального стейт-менеджера (Redux, MobX, Zustand) на текущем этапе для упрощения кодовой базы.

## 2. Принятое решение

Использовать классическую Single Page Application (SPA) архитектуру на следующем стеке:

- **База**: React 18+ с бандлером Vite.
- **Язык**: TypeScript со строгой типизацией.
- **Роутинг**: TanStack Router (обеспечивает строгую типизацию параметров и query-строки).
- **Работа с данными**: TanStack Query для управления серверным состоянием, кэшированием и инвалидацией данных.
- **Генерация GraphQL**: Использование `graphql-codegen` (с плагинами `typescript`, `typescript-operations`, `typescript-graphql-request`) для автоматической генерации типизированных запросов.
- **UI и стилизация**: Tailwind CSS в связке с shadcn/ui для быстрого создания доступных и кастомизируемых компонентов.
- **Интеграция Auth**: `@ory/elements` для готовых компонентов экранов аутентификации.
- **Управление состоянием**: Глобальный стейт-менеджер не используется (серверный стейт лежит в TanStack Query, стейт приложения — в URL через TanStack Router, локальный — в `useState/useReducer`).

## 3. Технические детали и механизмы

### Работа с GraphQL и TanStack Query

Так как вопрос кодогенерации для GraphQL был открыт, мы будем использовать библиотеку [GraphQL Code Generator](https://the-guild.dev/graphql/codegen).
Рабочий процесс:

1. Описываем запросы в `.graphql` файлах.
2. `graphql-codegen` парсит схему бэкенда и генерирует готовые типизированные функции для клиента (например, через `graphql-request`).
3. Эти функции оборачиваются в хуки `useQuery`/`useMutation` от TanStack Query.

### Отказ от стейт-менеджера

Архитектура строится на принципе разделения состояния:

- **Серверное состояние** (курсы валют, данные пользователя) — полностью управляется TanStack Query.
- **Стейт UI и фильтры** (выбранные параметры обмена, вкладки) — синхронизируются с адресной строкой (URLSearchParams) через возможности TanStack Router.
- **Локальный UI-стейт** (открытые модальные окна, состояние форм) — управляется локальными хуками React.

## 4. Рассмотренные альтернативы

### 4.1. Next.js (SSR / App Router)

- **Плюсы:** Решает проблемы SEO по умолчанию, удобная маршрутизация из коробки.
- **Минусы:** Избыточен для личного кабинета, где страницы скрыты за авторизацией (SEO не нужно). Усложняет инфраструктуру (нужен Node.js сервер вместо простой статики S3/CDN) и добавляет крутую кривую обучения серверным компонентам (RSC).
- **Решение:** Отклонено. Выбран Vite (SPA) для фокуса на клиентской логике, скорости сборки и упрощения деплоя.

### 4.2. Использование Redux Toolkit

- **Плюсы:** Единый стор для всего приложения, отличный DevTools, стандарт индустрии.
- **Минусы:** Много бойлерплейта. Дублирует функции кэширования, которые уже отлично решает TanStack Query.
- **Решение:** Отклонено. Мы принимаем решение обходиться без громоздкого менеджера состояний на ранних стадиях для сохранения простоты и скорости доставки функционала.

### 4.3. Apollo Client (вместо TanStack Query)

- **Плюсы:** Глубокая интеграция с GraphQL прямо из коробки, встроенный нормализованный кэш.
- **Минусы:** Достаточно "тяжелый", может быть сложен в тонкой настройке по сравнению с интерфейсом Query. Разработчики уже отлично владеют TanStack Query.
- **Решение:** Отклонено. Используем TanStack Query + легковесный GraphQL-клиент в угоду уже имеющейся экспертизе.

## 5. Последствия и риски

### Положительные (Pros)

- Высокая скорость разработки за счет использования знакомых команде инструментов (Vite, Tanstack).
- Минимальное время на написание CSS и базовых UI-компонентов благодаря Tailwind и shadcn/ui.
- Повышенная безопасность и типобезопасность на всех слоях (от маршрутизации до API).
- Уменьшение лишнего кода (отсутствие Redux, делегирование Auth-интерфейса `@ory/elements`).

### Отрицательные / Риски (Cons/Risks)

- **Риск:** Без стейт-менеджера сложный межкомпонентный обмен данными может превратиться в "prop drilling" или переизбыток React Context.
  - **Митигация:** Использовать композицию компонентов, держать бизнес-состояние в кэше Query или URL (через TanStack Router), применять Context API только локально для связанных компонентов.
- **Риск:** Настройка кодогенератора (`graphql-codegen`) в первый раз может потребовать времени на правильную конфигурацию плагинов для совместной работы с Vite.
  - **Митигация:** Заложить отдельную задачу / Spike на настройку инфраструктуры кодогенерации перед началом активной разработки продуктовых фич.
