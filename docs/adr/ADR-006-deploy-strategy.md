# ADR-006: Стратегия деплоя — Модульный монолит (Single Binary)

**Статус:** Accepted  
**Дата:** 2026-03-09  
**Автор:** kfreiman

## 1. Контекст

В рамках [ADR-005] была утверждена структура приложения на базе DDD-lite и изолированных Bounded Contexts. На текущем этапе проект носит демонстрационный характер (демо/MVP). Хранение состояния временно реализовано как in-memory репозитории, а взаимодействие между компонентами — как in-memory вызовы.

Важно отметить, что **in-memory инфраструктура не является финальной фичей или начальным бизнес-требованием**. Это исключительно промежуточный этап для ускорения разработки домена на старте. Архитектура приложения должна быть изначально спроектирована так, чтобы переход на внешние базы данных (PostgreSQL) и полноценные способы межсервисного взаимодействия (gRPC, NATS, Kafka) проходил бесшовно и без изменения бизнес-логики.

**Ограничения и требования:**

- Минимальные затраты времени на настройку CI/CD и инфраструктуры запуска на этапе демо/MVP.
- Логика разных контекстов (например, `billing`, `ordering`) должна иметь возможность взаимодействовать друг с другом.
- При использовании in-memory хранилищ данные живут только в рамках процесса ОС.
- Сохранить возможность легкого перехода на микросервисы в будущем, когда появятся реальные базы данных и Message Brokers.

## 2. Принятое решение

Использовать паттерн **Modular Monolith (Модульный монолит)**.
На этапе MVP всё приложение собирается в **один исполняемый файл (Single Binary)** и разворачивается как единственный процесс.

Разделение на контексты существует на уровне кодовой базы приложения (`internal/...`). Однако, каждый контекст **полностью изолирован** и, архитектурно, способен иметь собственный сервер и точку входа. Вся логика деплоя, инициализации инфраструктуры (In-Memory vs Postgres, Local Event Bus vs NATS/Kafka) и поднятия серверов (HTTP/gRPC/BFF) вынесена исключительно в слой `cmd/` (wireup-уровень). В будущем, для масштабирования, достаточно будет добавить новые папки в `cmd/` (например, `cmd/billing`, `cmd/ordering`), чтобы развернуть контексты как независимые сервисы, не меняя код в `internal/`.

## 3. Технические детали и механизмы

Единственной точкой входа будет `cmd/monolith/main.go`. В этом файле происходит:
1. Инициализация In-Memory (позже — реальных) репозиториев;
2. Сборка `Application` слоев каждого контекста;
3. Настройка компонентов IAM по [ADR-001] (взаимодействие с Kratos / Keto);
4. Поднятие BFF (согласно [ADR-002]) и роутов серверов;
5. Регистрация всего в едином процессе.

### 3.1 Взаимодействие (Синхронное и Асинхронное) и Консистентность

Согласно [ADR-001] и [ADR-002], в системе присутствуют механизмы синхронного и асинхронного взаимодействия. В рамках монолита они эмулируются in-memory, но реализуют те же интерфейсы:

- **Синхронное взаимодействие (RPC):** Согласно [ADR-002], S2S-вызовы проектируются под gRPC. В рамках монолита вместо сетевых gRPC-вызовов контексты общаются через прямые вызовы Go-интерфейсов. Интерфейсы (порты) спроектированы так, чтобы в будущем `cmd/` инжектировал сетевые gRPC-клиенты вместо in-memory заглушек.
- **Асинхронное взаимодействие (Events):** События публикуются в Event Bus. В MVP это In-Memory шина (каналы Gо), которая в будущем будет заменена на NATS или Kafka.
- **Интеграция с внешними сервисами IAM (Kratos / Keto):** Аутентификация и авторизация вынесены в Edge-слой (BFF / Gateway). BFF имеет **особую роль**: он является единой точкой входа в систему для внешних и внутренних клиентов (агрегация GraphQL), валидирует токены с IAM и уже в очищенном виде прокидывает запросы по контекстам.
- **Консистентность данных и транзакционность:** Поскольку база данных у каждого сервиса в будущем будет своя, распределенные транзакции не используются. Гарантия доставки (и обработки) событий решается через паттерн **Transactional Outbox**. Транзакционность ограничивается одним Bounded Context: при сохранении сущности в БД (через `adapters`), в той же транзакции создается запись исходящего события, которое затем асинхронно публикуется в Message Broker.

## 4. Рассмотренные альтернативы

### 4.1. Сразу чистые микросервисы (Один контекст = один процесс/контейнер)

**Суть:** Собирать каждый `internal/[context]` в отдельный бинарник и деплоить через k8s/docker-compose сразу же, используя gRPC.
**Почему отклонено:** В условиях отсутствия реальной БД (PostgreSQL) и брокера, контексты не смогут обмениваться данными без сложной реализации gRPC/HTTP клиентов между собой, а in-memory стейт каждого сервиса будет невидим для других. Огромные накладные расходы на поддержание инфраструктуры и CI/CD в самом начале (overengineering) для MVP, который призван быстро валидировать домен.

### 4.2. Традиционный монолит (отсутствие строгих границ)

**Суть:** Все компоненты в единой куче без интерфейсов и изоляции, общие слои, общая структура (ADR-005 не выполняется). 
**Почему отклонено:** Временный выигрыш в скорости обернется невозможностью распилить этот монолит на сервисы (ввиду сильной связности), когда in-memory этап завершится и придет время переходить к NATS, Postgres и микросервисной архитектуре. Выбранный нами Модульный монолит с DI обязывает соблюдать границы.

### 4.3. Serverless функции (AWS Lambda / Яндекс Cloud Functions)

**Суть:** Деплой каждого endpoint-а как независимой функции.
**Почему отклонено:** Чистая архитектура и DDD подразумевают некоторую тяжеловесность инициализации (DI, сборка слоев). Кроме того, In-memory состояние будет теряться при каждом завершении (cold Start) функции, что делает демо-проект нежизнеспособным. Вдобавок сложно оркестрировать инициализацию изолированных контекстов и In-Memory брокера.

## 5. Последствия и риски

### Положительные (Pros)

- **Zero-cost infrastructure:** Нужен только один `Dockerfile` размера около 10-15 MB (Alpine/Scratch) и простейший pipeline. Выполнение локально сводится к `go run cmd/monolith/main.go`.
- **Готовность к переходу на микросервисы:** Логика деплоя (wireup) ограничена директорией `cmd/`. При переходе на MMSA достаточно создать `cmd/ordering/main.go` и заменить in-memory адаптеры на Postgres/gRPC клиенты. `internal/` останется нетронутым.
- **Идеально для MVP:** Все контексты делят общее адресное пространство ОС. In-memory шина событий будет работать с нулевой задержкой.

### Отрицательные / Риски (Cons/Risks)

- Разработчики могут по лени нарушить ADR-005 и начать импортировать In-memory репозитории чужих контекстов напрямую, раз уж всё в одном процессе.
- **Митигация:** Использовать средства контроля или линтеры (например, `go-arch-lint` или `depguard`), которые строго запретят импорты между папками `internal/context_a/...` и `internal/context_b/...`. Взаимодействие должно идти строго через порты (interfaces в слое App/Adapters) или шину.
